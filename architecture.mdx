Datology is deployed into the customer's AWS account. The deployment has two
layers: AWS infrastructure and the Kubernetes product stack.

## Infrastructure layer

The CloudFormation deployment creates or connects to the AWS resources Datology
needs:

- VPC, subnets, security groups, and VPC endpoints
- EKS cluster and access entries
- IAM roles and policies for installers, workers, Spark, Ray, and the CLI runner
- S3 buckets for assets, logs, code artifacts, and release assets
- DNS records and certificates for service endpoints
- EC2 instances for installation and CLI operations

## Kubernetes product stack

The product stack runs on EKS and installs the services used by curations:

- Workflow orchestration
- PostgreSQL databases for orchestration and asset tracking
- Asset tracker service and metadata schema
- Spark Operator
- KubeRay
- Karpenter, YuniKorn, and Kueue for scheduling
- Monitoring and logging integrations
- AWS load balancer controller

## Curation path

```text
Customer operator
  -> datology CLI on the CLI runner
  -> recipe, dataset, version, and output path
  -> registered recipe implementation
  -> Spark and Ray jobs on EKS
  -> object storage artifacts
  -> asset tracker metadata
```

The `datology` CLI is the main customer entry point. Use it for dataset
registration, curation launches, status, cancellation, and asset inspection.

`ddb` is an advanced debug surface for deeper run, Spark, Ray, task, and log
inspection. Use it with Datology support. This is where you may see Flyte,
Spark, Ray, Kubernetes, and asset-tracker terms in logs and status output.
Normal customer workflows should stay on `datology`.
