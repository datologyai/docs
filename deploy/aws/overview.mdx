Datology is deployed and operated in your AWS account. Datology can help scope
and troubleshoot the install, but the stack is customer-owned.

## Two deployment layers

The install has two layers.

| Layer | What it creates |
| --- | --- |
| AWS infrastructure | VPC or existing network integration, EKS, IAM roles, S3 buckets, DNS, certificates, installer infrastructure, and the CLI runner. |
| Kubernetes product stack | Workflow orchestration, databases, asset tracker, Spark Operator, Ray support, scheduling components, autoscaling components, and monitoring. |

## Released install path

The public deployment path is the released stack entry point:

```bash
python3 download_and_deploy_datology.py
```

Before running it, review `cluster_config.yaml`. It controls the size and shape
of the resources the stack will create. The downloader retrieves the released
deployment scripts and runs the installer. The installer looks up released
templates and artifacts, creates the CloudFormation parent stack, installs the
Kubernetes product stack, and prepares the CLI runner.

## Deployment outputs

Keep these values after installation:

- CloudFormation stack name
- EKS cluster name
- CLI runner instance ID and access method
- `DATOLOGY_CONFIG_PATH`
- `FLYTECTL_CONFIG`
- `ASSET_TRACKER_PROD_URI`
- `KUBECONFIG`
- Asset, log, code-artifact, and model/artifact bucket names
- Runtime image tag
- Registered recipe version

Most CLI commands need the generated config file, workflow endpoint config, and
asset tracker connection. Operator commands also need Kubernetes access.

## Recommended order

1. Review [AWS prerequisites](/deploy/aws/prerequisites).
2. Choose a [network and access](/deploy/aws/network-and-access) pattern.
3. Confirm [storage and bucket](/deploy/aws/storage-and-buckets) requirements.
4. Run the [install](/deploy/aws/install).
5. Connect to the [CLI runner](/deploy/aws/cli-runner).
