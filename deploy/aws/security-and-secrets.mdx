Datology runs in your AWS account and uses AWS-native identity and secret
storage.

## IAM areas

The deployment creates or uses roles and policies for:

- CloudFormation stack creation
- Helm installer operations
- CLI runner operations
- EKS node groups
- Flyte workers
- Spark jobs
- Ray jobs
- Indexed Kubernetes jobs, when enabled
- AWS load balancer controller

Use `PERMISSION_BOUNDARY_POLICY_ARN` when your AWS organization requires a
permissions boundary.

## Bucket access

Datology grants worker roles access to deployment buckets and customer artifact
buckets. Use these variables for extra customer-owned data:

```bash
export ADDITIONAL_READ_ONLY_BUCKET_ARNS="arn:aws:s3:::source,arn:aws:s3:::source/*"
export ADDITIONAL_READ_WRITE_BUCKET_ARNS="arn:aws:s3:::target,arn:aws:s3:::target/*"
```

Read-write access should be scoped to output and staging locations.

## Database secrets

The asset tracker and workflow orchestration databases use Kubernetes secrets
and AWS Secrets Manager access patterns created by the deployment.

The CLI runner derives `ASSET_TRACKER_PROD_URI` from the deployed asset tracker
database service and secret.

## Model and Hugging Face tokens

Some dependency commands need access to model registry or Hugging Face tokens.
Document the customer-approved secret names and rotation process before making
those commands broadly available.

## Network access

Restrict SSH and API access with:

```bash
export ALLOWED_SSH_CIDR="<trusted-cidr>"
export ALLOWED_EKS_API_CIDR="<trusted-cidr>"
```

For private deployments, also validate the required VPC endpoints before
launching curations.
